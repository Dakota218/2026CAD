# 程式碼優化演進過程

## 1. 文件目的

本文件整理本專案從初版 optimizer 到目前版本的程式碼優化演進過程。重點不是只列出修改紀錄，而是說明每一階段遇到的問題、採取的改善方向，以及這些修改如何逐步形成目前的 useful skew timing fixing flow。

本文件可放在正式報告的「方法演進」、「實作歷程」或附錄中，協助說明程式不是一次完成，而是依據 testcase 結果、checker 回饋與 runtime bottleneck 逐步修正。

資料來源包含：

| 資料來源 | 用途 |
|---|---|
| `D_testcase_20260601/CODEX_CHANGELOG.md` | 早期至中期程式修改紀錄 |
| `D_testcase_20260601/optimization_notes.md` | 後期實驗、incremental STA、target-delay 策略整理 |
| `D_testcase_20260601/optimization_log.md` | 單次 optimizer run 的 phase / candidate / score terms 紀錄 |

| 最終 source code | `main.cpp`、`parser.cpp`、`timing_engine.cpp`、`optimizer.cpp` |

## 2. 整體演進摘要

本專案的 optimizer 大致經過七個主要階段：

| 階段 | 核心問題 | 主要改善 |
|---|---|---|
| 1. 基礎可執行版本 | optimizer 需要在時間內結束，並能產生合法輸出 | 加入時間保護、candidate 數量限制、基本 greedy flow |
| 2. Timing model 修正 | 內部 timing 與 checker 結果不一致 | 修正 setup / hold slack 公式，修正 clock period parsing |
| 3. WNS-first greedy | 初版只做局部修正，對 worst slack 改善不足 | 依 top violated paths 產生候選，優先修 SS WNS |
| 4. Buffer chain 與 TNS/NVP cleanup | 單點 leaf insertion 難以同時改善多條 path | 支援多 buffer chain、加入 capture sink 聚合與 internal insertion |
| 5. Robust score 與 phase scheduler | 只看單一指標容易犧牲其他項 | 使用 normalized terms、robust score、phase-specific acceptance |
| 6. Global target skew 與 beam search | local greedy 容易被短期最佳解限制 | 建立 sink target delay、導入 sequence / beam-style search |
| 7. Incremental STA 與最終整理 | 精確評估 candidate 太慢 | 建立 affected path index，只更新受影響 timing，降低評估成本 |

最後形成的版本以 `TimingEngine` 負責 timing/area/合法性評估，以 `Optimizer` 負責 phase scheduling、candidate generation、score-term objective 與 best solution selection。

## 3. 階段一：基礎 optimizer 與 runtime 保護

最早的問題是 optimizer 可能在大型 testcase 中花太久，或在候選數量過多時無法於 contest 限制內完成。因此第一步先建立基本安全框架：

| 修改項目 | 說明 |
|---|---|
| 加入 optimizer 內部時間保護 | 使用 `std::chrono::steady_clock` 計時，到達時間限制即停止並回傳目前最佳 clock tree |
| 加入 candidate 上限 | 避免每輪枚舉過多 resize / insertion candidate |
| 修正 iterator / reference 失效風險 | 修改 clock tree 前先收集 node name，避免邊走訪邊修改造成錯誤 |
| 加入 debug output | 印出初始 score、candidate 數量、評估數、elapsed time、最終 metrics |

這一階段的重點是讓程式「穩定跑完」。雖然最佳化品質還不夠好，但已具備後續實驗需要的安全外殼。

## 4. 階段二：Timing model 與 parser 修正

後續發現 optimizer 內部 timing model 需要與 checker 對齊，否則候選 move 在內部看似改善，實際 checker 可能不一致。這一階段主要修正 timing 公式與輸入解析。

### 4.1 Slack formula 修正

`timing_engine.cpp` 中的 slack 公式修正為：

```text
SS setup slack = Tclk - 0.08 * Tclk - data_delay + skew
FF hold slack  = data_delay - 0.05 * Tclk - skew
```

其中 skew 可理解為 capture clock arrival 與 launch clock arrival 的差。對 setup 而言，capture clock 較晚通常有利；對 hold 而言，過大的 capture-late skew 可能不利。因此這個公式修正是後續 direction-aware useful skew 的基礎。

### 4.2 Clock period parsing 修正

`optimization_notes.md` 記錄 delay report 使用格式：

```text
Clock Period : 0.3
```

舊 parser 只匹配 `Clock Period:`，導致 `clock_period` 可能維持 0，使 optimizer timing model 與 checker 不一致。後來修正 parser，使其能接受冒號前有空白的格式。這個修改雖小，但對 timing calculation 正確性非常關鍵。

## 5. 階段三：WNS 優先的 repeated greedy

在 timing model 修正後，下一個瓶頸是 worst slack 改善不足。若 optimizer 只看總分或局部 area，很容易改善許多小 violation，卻沒有處理最差的 SS WNS。因此程式改成 WNS-first repeated greedy。

主要修改包含：

| 修改 | 目的 |
|---|---|
| 分析 top violated paths | 初始輸出 top SS setup / FF hold violated paths，觀察最嚴重路徑 |
| 依目前 violation 產生 candidate | 每輪根據最新 timing 狀態重新產生候選 |
| setup capture side 加 delay | SS setup violation 優先延後 capture FF |
| hold launch side 加 delay | FF hold violation 優先延後 launch FF |
| ancestor priority 傳遞 | 讓 resize / internal insertion 可以影響 shared subtree |
| WNS-first scoring | 先比較最差 WNS，再比較 WNS sum、TNS、NVP、area |

這一版的核心概念是：不要只平均改善所有 path，而是先讓最差 path 有機會被修掉。從早期 sanity run 觀察，testcase2 的 SS WNS 有明顯改善，FF hold violation 也能下降到 0，表示方向式 useful skew 策略有效。

## 6. 階段四：Buffer chain candidate 與 TNS/NVP cleanup

只插入單顆 buffer 時，可提供的 delay granularity 有限；而且若每次只修一個 FF sink，對 TNS/NVP 的改善速度也慢。因此接著加入 buffer chain 與 TNS/NVP cleanup。

### 6.1 Multi-buffer chain

原本 leaf insertion 逐步擴充為 2-buffer chain，再到 WNS repair 中支援 3-buffer chain。格式如下：

```text
parent -> NEW_BUF_A -> NEW_BUF_B -> NEW_BUF_C -> sink
```

這樣可以用多顆 buffer 組合出更接近 `need_delay = -slack` 的 clock delay。`applyMove()` 也被更新為可套用多顆 buffer chain，仍維持新 buffer 使用 `NEW_BUF_X` 命名，並保留原 parent-child edge 的插入關係。

### 6.2 TNS/NVP cleanup

WNS 修到一定程度後，若仍只盯著最差 slack，可能忽略大量中等 violation。因此加入 TNS/NVP cleanup：

| Cleanup 方法 | 說明 |
|---|---|
| 分析 top 1000 SS violated paths | 擴大觀察範圍，不只看 top 10 |
| 依 capture FF 聚合 | 統計同一 capture sink 影響多少 violated paths |
| 使用 `count`、`total_neg_slack`、`worst_slack` 排序 | 找出對 TNS/NVP 貢獻大的 sink |
| 對 ancestor 傳遞 priority | 讓 internal insertion 可以一次影響一群 capture sinks |

這一階段的目標是從「修最差單一路徑」進一步變成「改善一批 violation」。對大型 testcase 來說，這比單點修補更有機會降低 TNS 與 NVP。

## 7. 階段五：Robust score 與接受條件改版

隨著候選 move 變多，若接受條件太單純，就可能發生 setup 改善但 hold 大幅惡化，或 timing 小幅改善但 area 增加太多的情況。因此程式改成使用 normalized score terms 與 robust score。

Normalized terms 定義為：

```text
SS_TNS_term = 1 - Modified_SS_TNS / Original_SS_TNS
SS_WNS_term = 1 - Modified_SS_WNS / Original_SS_WNS
FF_TNS_term = 1 - Modified_FF_TNS / Original_FF_TNS
FF_WNS_term = 1 - Modified_FF_WNS / Original_FF_WNS
AREA_term   = 1 - Modified_Area / Original_Area
```

程式後來在 `TimingEngine::computeScoreTerms()` 中加入 timing_score：

```text
timing_score = 4.0 * SS_WNS_term
             + 3.0 * SS_TNS_term
             + 2.0 * FF_WNS_term
             + 1.5 * FF_TNS_term
             + 0.5 * AREA_term
```

此權重反映實作策略：setup WNS / TNS 優先，其次 hold，最後 area。`evaluateRobustScore()` 也使用多組 alpha 權重估計 robust score，避免只對單一 scoring assumption 過度最佳化。

同時，candidate acceptance 加入多種 guard：

| Guard | 目的 |
|---|---|
| setup degradation guard | 避免 SS WNS/TNS 明顯變差 |
| hold degradation guard | 避免 FF hold 被 setup move 破壞 |
| area hurt timing guard | 面積增加時必須換到足夠 timing 改善 |
| normalized damage guard | 防止某一項 score term 被過度犧牲 |

這使 optimizer 從單純 greedy 逐步變成多目標、有保護條件的搜尋流程。

## 8. 階段六：Phase scheduler、group insertion 與回退

曾經嘗試加入較完整的 time-budget phase scheduler：

| 時間窗 | Phase | 目的 |
|---:|---|---|
| 0-180 sec | `SETUP_WNS` | 優先修 setup WNS |
| 180-420 sec | `TNS_CLEANUP` | 加入 top 1000 SS violations 聚合 |
| 420-510 sec | `GROUP_INSERT` | 加入 shared ancestor edge insertion |
| 510-540 sec | `HOLD_AREA` | 修 hold 並做 area cleanup |

其中 group insertion 會分析 setup-critical capture FF 的 shared ancestor edge，希望一次 delay 一整個 subtree，降低 leaf-only insertion 的 area 成本。這個方向在概念上合理，但實驗後發現 phase scheduler 與 shared edge candidate 仍需要更多調參，因此曾依需求回退到較穩定版本。

這次回退保留了已被證明有用的功能，例如 robust score、setup 優先、hold guard、WNS repair chain、TNS/NVP cleanup；移除較不穩定的 time-budget scheduler 與 shared ancestor 實驗。這代表開發過程並不是所有新增功能都保留，而是以 checker 結果和穩定性為準。

## 9. 階段七：Global target skew 與 beam / sequence search

後期優化開始嘗試更 global 的方向。單純 local greedy 容易只看到當下最佳 move，但 clock skew 是全域耦合問題，同一個 sink 同時可能出現在許多 setup / hold paths 中。因此加入 target-skew 概念。

### 9.1 Global sink target solver

`optimization_notes.md` 中整理的 intended flow 如下：

1. 使用目前所有 setup / hold violations 估計每個 sink 的 target delay delta。
2. 正 target 表示該 sink / subtree 應增加 clock delay。
3. 負 target 表示該 sink / subtree 應減少 clock delay。
4. 將正 target 對應到 leaf / internal insertion candidate。
5. 將負 target 對應到 ancestor resize priority，使其偏向 faster cell。
6. 最後仍由 exact timing evaluation 決定 candidate 是否接受。

這不是完整 ILP，但比純 local greedy 更全域，因為每個 sink 的方向是從多條 path 累積出來的。目前 final code 中可看到 `TargetSkewGuide` 與 `buildTargetSkewGuide()`，即是此方向的具體實作。

### 9.2 Beam / sequence search

後期 cleanup 與 group phase 不再只做 one-step greedy，而是使用 bounded beam-style sequence search：

| 步驟 | 說明 |
|---|---|
| First layer | 先評估一批第一步 move |
| Keep promising states | 保留數個有潛力的中間狀態 |
| Expand second move | 從中間狀態再展開下一步 |
| Select best legal sequence | 在 exact-evaluation budget 內選擇最佳合法 sequence |

這樣可以處理某些「單步看起來普通，但連續兩步後效果較好」的情況。對 clock tree 修改而言，這很重要，因為插入或 resize 後會改變後續 candidate 的 fanout、delay 與 affected paths。

## 10. 階段八：Incremental STA 加速

候選 move 數量增加後，最大 runtime bottleneck 變成 timing evaluation。如果每個 candidate 都完整重算所有 path，對大型 testcase 會太慢。因此後期重點轉向 incremental timing engine。

`TimingEngine` 現在會建立以下索引：

| Index / cache | 用途 |
|---|---|
| node 到 descendant sink IDs | 找出某個 subtree 修改會影響哪些 sinks |
| sink 到 setup launch/capture paths | 快速找到受影響 setup paths |
| sink 到 hold launch/capture paths | 快速找到受影響 hold paths |
| cached path slack state | 避免每次從頭重建 path slack |
| negative slack multiset | 快速更新 WNS/TNS/NVP |

`TimingEngine::evaluateMove()` 評估 `ResizeBuffer` 或 inserted-chain move 時，只計算 affected subtree delay delta，並只更新 incident paths。評估完會 rollback 暫時 slack 更新，因此 optimizer 可以把它當成快速 what-if evaluator。

這個修改讓後續 candidate search 能使用更多候選與更精細的 acceptance rule，而不至於每輪都完整重算所有 timing path。

## 11. 最終程式架構整理

目前程式可分成四個主要模組：

| 模組 | 角色 |
|---|---|
| `parser.cpp` / `parser.h` | 讀取 clock tree、buffer library、SS/FF delay report，輸出 modified structure |
| `timing_engine.cpp` / `timing_engine.h` | 計算 SS/FF timing metrics、area、score terms，提供 incremental candidate evaluation |
| `optimizer.cpp` / `optimizer.h` | 產生候選 move、控制 phase、決定接受條件、選出 best tree |
| `main.cpp` | 串接 parser、timing engine、optimizer，輸出 final summary |

最終 optimizer 的主要流程如下：

```text
Read input
  ↓
Build initial TimingEngine
  ↓
Analyze original timing and area
  ↓
Build adaptive phase schedule
  ↓
For each phase:
    analyze current violations
    build target skew guide
    generate candidate moves
    trim and rank candidates
    evaluate by incremental TimingEngine
    accept phase-safe best move
  ↓
Compare mode states
  ↓
Output selected best clock tree
```

## 12. 最終結果觀察

根據 `testcase00_04_results.md`，目前結果如下：

| Testcase | Setup 改善 | Hold 改善 | Area 變化 | 觀察 |
|---|---|---|---|---|
| testcase0 | SS WNS -0.5286 → -0.3385；SS TNS -1134.1601 → -798.6013 | FF WNS -0.1085 → -0.0279；FF TNS -3.0381 → -0.4664 | +0.05% | 大型 violation case，timing 大幅改善但付出少量 area |
| testcase1 | SS WNS -0.4023 → -0.2017；SS TNS -252.4482 → -76.0461 | FF WNS/TNS/NVP 歸零 | -8.60% | hold 修復成功，area 明顯下降 |
| testcase2 | SS WNS -0.3020 → -0.0932；SS TNS -98.6976 → -19.6859 | FF WNS/TNS/NVP 歸零 | -18.27% | 最成功的 area recovery case |
| testcase3 | SS WNS -0.4594 → -0.2249；SS TNS -374.2329 → -228.7882 | FF WNS/TNS/NVP 歸零 | -2.50% | hold 完全修復，setup 仍有剩餘 violation |
| testcase4 | SS WNS -0.5938 → -0.4707；SS TNS -2277.6896 → -1473.3182 | FF WNS -0.1219 → -0.0650；FF TNS -6.9849 → -3.0061 | +0.07% | 最大 testcase，仍是 setup/hold timing bottleneck |

整體而言，程式演進後最明顯的成果是 hold repair 能在中小型 testcase 中歸零，setup TNS/WNS 在所有 testcase 均改善。area 部分則依 testcase 特性不同：較小 testcase 可透過 downsize 和 cleanup 降低面積；較大 testcase 則需要額外 buffer 修 timing，因此 area 小幅增加。

## 13. 演進過程中的取捨

本專案的優化過程有幾個主要取捨：

| 取捨 | 說明 |
|---|---|
| WNS vs TNS/NVP | 只修 WNS 會忽略大量 violating paths；只修 TNS 又可能留下 worst slack |
| Setup vs Hold | 延後 capture clock 有利 setup，但可能傷害 hold，需要 direction-aware guard |
| Timing vs Area | 插入 buffer 常可改善 timing，但會增加 area；downsize 可省 area，但方向錯會傷 setup |
| Candidate richness vs runtime | 更多 chain / resize / internal insertion 增加找到好解的機會，也增加評估成本 |
| Local greedy vs sequence search | Local greedy 快但短視；beam / sequence search 較慢但可找到連續 move 的效果 |
| New feature vs stability | Phase scheduler / group insertion 有潛力，但若不穩定就先回退保留可靠功能 |

## 14. 可放入正式報告的簡短版本

若正式 Word 報告篇幅有限，可使用下列精簡敘述：

> 本專案的 optimizer 由初版 repeated greedy 逐步演進為多 phase、direction-aware 的 useful skew optimizer。早期版本先加入 runtime guard、candidate 上限與 checker 對齊的 timing formula，確保程式能在限制時間內產生合法輸出。接著，為改善 setup worst slack，候選產生改為依照 top violated paths 對 setup-critical capture side 加 delay、對 launch side 避免加 delay；hold repair 則採相反方向。後續加入 multi-buffer chain、TNS/NVP cleanup、internal insertion 與 normalized score terms，使 optimizer 不只修單一路徑，也能改善大量 violation paths。最後，為降低候選精確評估成本，`TimingEngine` 建立 affected sink/path index，使用 incremental STA 只更新受影響 path，讓 optimizer 可以在有限 runtime 內評估更多候選。最終結果顯示，五個 testcase 的 SS setup 均改善，三個 testcase 的 FF hold violation 完全修復，並在部分 testcase 中成功降低 buffer area。
