## 2. 題目簡介

本題是 ICCAD 2026 CAD Contest Problem D，主題為 post-CTS 階段的 timing fixing by useful skew。題目給定一棵已建立的 clock tree、buffer library，以及 SS corner 與 FF corner 的 timing delay report。參賽程式需要修改 clock tree structure，使不同 FF sink 的 clock arrival time 產生適當偏移，藉由 useful skew 改善 setup 與 hold timing。

輸入資料主要包含：

| 輸入檔 | 說明 |
|---|---|
| `clk_tree.structure` | 原始 clock tree 拓樸，包含 root、buffer、FF sink 與 parent-child 關係 |
| `buf.lib` | 可使用的 buffer cell type、面積、max fanout、SS/FF delay table |
| `SS_delay.rpt` | SS corner 下的 data path delay 與 clock period，用於評估 setup timing |
| `FF_delay.rpt` | FF corner 下的 data path delay 與 clock period，用於評估 hold timing |

輸出為修改後的 `modified_clk_tree.structure`。合法操作包含在原本 parent-child edge 中插入新 buffer chain，新 buffer 命名為 `NEW_BUF_X`；也可以 resize 既有 buffer 的 cell type。輸出必須保留原本節點，不可刪除或重新命名既有節點，也不可破壞原始節點順序與 parent-child 合法性。所有 buffer 仍需符合 library 中定義的 max fanout 限制。

最佳化目標是同時改善 SS setup WNS/TNS、FF hold WNS/TNS，並盡量減少或控制 buffer area。評分概念可用 normalized score terms 表示：

```text
SS_TNS_term = 1 - Modified_SS_TNS / Original_SS_TNS
SS_WNS_term = 1 - Modified_SS_WNS / Original_SS_WNS
FF_TNS_term = 1 - Modified_FF_TNS / Original_FF_TNS
FF_WNS_term = 1 - Modified_FF_WNS / Original_FF_WNS
AREA_term   = 1 - Modified_Area / Original_Area
```

因為原始 timing violation 的 TNS/WNS 多為負值，若修改後 violation 越接近 0，對應 term 會越接近 1。Area term 則表示面積是否比原始設計更小；若面積增加，AREA_term 會變成負值。此題的困難點在於 setup、hold、area 彼此會互相牽制：延後某些 capture clock 可以改善 setup，但若方向用錯，可能造成 hold 惡化；縮小 buffer 可以省面積，也可能增加 delay，因此必須根據路徑方向判斷是否有利。

## 4. 方法介紹

### 4.1 Overall strategy

本專案的主策略是利用 clock tree 的 buffer insertion 與 buffer resizing 調整各 FF sink 的 clock arrival time。對 setup timing 而言，若 capture FF 的 clock arrival 比 launch FF 晚，可以讓資料有較多時間到達，因此有機會改善 SS setup slack。對 hold timing 而言，過大的 capture-launch skew 可能讓 hold 變差，因此 hold 修復時會反過來偏向延後 launch side 或加速 capture side。

程式入口在 `main.cpp`。它使用 `Parser::parseStructure()`、`Parser::parseLib()`、`Parser::parseDelayRpt()` 讀入 clock tree、buffer library、SS/FF delay report，再建立 `TimingEngine` 計算原始 TNS、WNS、NVP 與 area。接著呼叫 `Optimizer::runOptimization()` 產生最佳化後的 clock tree，最後由 `Parser::writeOutput()` 輸出 `modified_clk_tree.structure`。

最佳化核心在 `optimizer.cpp`。目前程式不是單一貪婪策略，而是分成多個 phase。`buildAdaptivePhases()` 目前設定 89 秒內的主要階段：

| Phase | 目的 | 時間窗 | Exact evaluation limit |
|---|---:|---:|---:|
| `setup_WNS_breakthrough directional_seed` | 先用方向感的 downsize / delay seed 找 setup WNS 突破點 | 0.0-18.0 sec | 2048 |
| `setup_worst_path_surgery` | 集中處理最差 setup path | 18.0-83.0 sec | 1200 |
| `setup_TNS_cleanup` | 清理 setup TNS / NVP | 83.0-83.1 sec | 128 |
| `strong_hold_repair` | 修復 FF hold violation | 83.1-89.0 sec | 1536 |
| `balanced_area_cleanup` | 在 timing 足夠安全時做面積整理 | 88.0-89.0 sec | 512 |

實際執行時，程式會在每個 phase 重新分析目前 violation，產生候選 move，使用 timing engine 做精確評估，通過 phase-specific acceptance rule 後才套用。這樣可以避免單純依估計 delay 做決策而破壞 timing。

### 4.2 Normalized score-term objective

程式使用 `TimingEngine::computeScoreTerms()` 計算 normalized score terms：

```text
SS_TNS_term = 1 - Modified_SS_TNS / Original_SS_TNS
SS_WNS_term = 1 - Modified_SS_WNS / Original_SS_WNS
FF_TNS_term = 1 - Modified_FF_TNS / Original_FF_TNS
FF_WNS_term = 1 - Modified_FF_WNS / Original_FF_WNS
AREA_term   = 1 - Modified_Area / Original_Area
```

若某一個 timing term 越接近 1，代表該項 violation 越接近被修完。若 FF WNS 或 TNS 已修到 0，該項 term 會接近或達到 1。Area term 若為正，表示面積下降；若為負，表示面積比原始設計增加。

程式中另有 `evaluateRobustScore()`，把 normalized terms 整理為 setup sum、hold sum、area，並使用不同 alpha 權重估計 robust score。`TimingEngine::computeScoreTerms()` 也定義一個 timing_score：

```text
timing_score = 4.0 * SS_WNS_term
             + 3.0 * SS_TNS_term
             + 2.0 * FF_WNS_term
             + 1.5 * FF_TNS_term
             + 0.5 * AREA_term
```

從權重可看出，程式優先處理 setup WNS 與 setup TNS，其次是 FF hold，最後才是 area。這符合本題特性：如果 setup/hold violation 還很嚴重，單純追求面積下降不一定能提高總分。程式註解也說明 AREA 是小型 penalty / tie-breaker，不應在 timing 仍有大幅進步空間時否決有效 timing move。

### 4.3 Downsize-for-delay strategy

Buffer resizing 不只用來降面積，也可以調整 clock delay。若把高 drive strength buffer 改成較小 cell，通常 area 會下降，但 delay 可能上升。當這個操作發生在 setup-critical capture side 時，capture clock 被延後，可能改善 setup WNS/TNS。因此程式設計了 `PhaseKind::DownsizeForDelay` 與 `addDownsizeForDelayMoves()`，用於尋找「面積下降且 timing_score 仍改善」的 resize move。

不過 downsizing 也有風險。如果在 setup-critical launch side 增加 clock delay，launch clock 變晚，反而會壓縮 setup 可用時間。因此程式用 `CandidateDirection` 標示 `CaptureSide`、`LaunchSide`、`Neutral`，並透過 `buildTargetSkewGuide()` 建立每個 sink / node / edge 的目標方向。`generateCandidateMoves()` 在產生 resize candidate 時，會根據 node priority 判斷需要加 delay 或減 delay；若 priority 為正，候選傾向增加 delay，若 priority 為負，候選傾向減少 delay。最後仍由 `TimingEngine::evaluateResizeMove()` 做精確評估，不只依估計值決定。

### 4.4 Setup repair

Setup repair 主要針對 SS corner。`analyzeTiming()` 會產生目前 SS violation list，`generateCandidateMoves()` 再依 phase 產生不同候選：

| Setup 相關功能 | 對應函式 / 結構 | 說明 |
|---|---|---|
| Worst path surgery | `addWorstPathSurgeryMoves()`、`PhaseKind::WorstPathSurgery` | 針對最差 setup path 建立 targeted candidate |
| WNS repair | `addWnsRepairMoves()`、`setupWnsFocusedScore()` | 對 SS WNS 給較高權重，優先改善最差 slack |
| TNS/NVP cleanup | `addTnsCleanupFocus()`、`tnsCleanupPriority()` | 聚合同一 capture sink 或 ancestor 影響的多條 violation path |
| Target skew guide | `buildTargetSkewGuide()` | 由 setup / hold path 推估 sink 需要增加或減少的 clock delay |
| Chain candidate | `buildChainLookup()`、`addChainMovesForEdge()` | 以 buffer library 建立可用 chain，再選擇接近目標 delay 的插入鏈 |

對 setup path 而言，程式的方向是「delay capture side、speed up launch side」。在 `generateCandidateMoves()` 中，SS violation 的 capture FF 會得到正 priority，並透過 `addAncestorsForFocus()` 把權重傳到 ancestor node / edge；launch FF 則會得到負 priority，避免在 launch side 加 delay。候選 move 包含：

| Move kind | 說明 |
|---|---|
| `InsertLeaf` | 在 parent 到 FF sink 的 edge 插入 buffer chain |
| `InsertInternal` | 在 internal edge 插入 buffer chain，影響整個 subtree |
| `ResizeBuffer` | 將既有 buffer resize 成其他合法 cell type |

接受條件由 `acceptableForPhase()` 控制。例如 `WorstPathSurgery` 要求 SS WNS 直接改善，或至少剝離 worst cluster 並改善 setup TNS；`WnsRepair` 會用 `setupWnsFocusedScore()` 判斷候選是否確實幫助 WNS；`TnsCleanup` 則重視 SS TNS 或 NVP 改善，同時用 `setupProtected()` 避免 WNS/TNS 明顯倒退。

### 4.5 Hold repair

Hold repair 主要針對 FF corner，在 `PhaseKind::HoldRepair` 中處理。Hold violation 的方向和 setup 不同：若 FF hold 太差，通常需要延後 launch side 或加速 capture side，以降低不利 skew。`buildTargetSkewGuide()` 在處理 FF paths 時，會把 hold violation 的 launch FF target 往正方向調整，capture FF target 往負方向調整。`generateCandidateMoves()` 在 hold phase 也會對 launch / capture 兩側設定相反的 node priority。

`acceptableForPhase(PhaseKind::HoldRepair, ...)` 要求 `FF_TNS_term + FF_WNS_term` 比目前更好，同時用 `setupProtected()` 保護 setup 不被大幅破壞。`betterCandidateForPhase()` 在 hold phase 先比較 `FF_WNS_term`，再比較 `FF_TNS_term`，代表 hold repair 階段會先看最差 hold slack 是否改善，再看整體 hold TNS。

從最終 testcase 結果看，testcase1、testcase2、testcase3 的 FF hold 已修到 WNS = 0、TNS = 0、NVP = 0；testcase0 與 testcase4 仍有剩餘 hold violation，但 testcase0 的 FF TNS 從 -3.0381 改善到 -0.4664，FF WNS 從 -0.1085 改善到 -0.0279，已大幅下降。

### 4.6 Area cleanup

Area cleanup 的策略是保守的。程式有 `PhaseKind::AreaRecovery` 與 `addAreaRecoveryMoves()`，也在 `GroupInsertion` / `balanced_area_cleanup` 中加入安全整理。不過 `acceptableForPhase()` 對 `AreaRecovery` 設了很高的 timing gate：setup 與 hold normalized terms 必須先達到一定水準，才允許單純為了 area recovery 接受 move。

此設計的原因是，本題 area 只是評分項之一，若 timing violation 還很嚴重，過早縮小 buffer 可能會讓 setup 或 hold 惡化。程式在 cleanup phase 會要求：

| 保護條件 | 說明 |
|---|---|
| `setupProtected()` | SS WNS/TNS 不可超過容許範圍惡化 |
| `holdProtected()` | FF WNS/TNS 不可超過容許範圍惡化 |
| fanout legality | resize 後 fanout 不可超過 cell max fanout |
| NVP 不增加 | Area recovery 中要求 setup/hold NVP 不變差 |

從 checker 結果看，testcase1、testcase2、testcase3 成功降低 area，其中 testcase2 area 下降 18.27%；testcase0 與 testcase4 則略微增加 area，代表這兩個 testcase 的 timing improvement 需要付出少量 buffer area。

### 4.7 Candidate evaluation and acceptance

候選 move 的評估流程如下：

1. `generateCandidateMoves()` 依 phase 產生候選。
2. `trimCandidateMoves()` 對候選排序與裁切，避免候選數過大。
3. `searchPhaseCandidates()` 分批做精確評估。
4. `evaluateCandidateMove()` 將 optimizer move 轉成 `TimingMove`。
5. `TimingEngine::evaluateResizeMove()` 或 `TimingEngine::evaluateInsertChainMove()` 做 incremental timing evaluation。
6. `acceptableForPhase()` 檢查該 phase 的接受條件。
7. `betterCandidateForPhase()` 選出同一批候選中的最佳 move。
8. `applyMove()` 真正修改工作中的 clock tree。
9. 每個 phase 結束後建立 `ModeState`，最後由 `betterModeState()` 選出輸出解。

`TimingEngine` 是降低 runtime 的關鍵。它在初始化時建立 descendant sink map、setup/hold launch/capture path index，以及 negative slack multiset。評估 resize 或 insertion 時，`TimingEngine::evaluateMove()` 只更新受影響 subtree 與相關 path 的 slack，再回滾暫存變更，因此比每次完整重算所有 path 更快。

合法性方面，`applyMove()` 對 resize 會檢查 node 是否存在、不可 resize ROOT 或 sink、fanout 不可超過新 cell 的 max fanout；對 insertion 則檢查 chain length、cell type 是否存在、新 buffer 命名是否唯一，並用 `NEW_BUF_X` 接在原 parent-child edge 中間。最後會呼叫 `checkFanoutLegal()` 確認整棵樹仍合法。

## 7. 結果分析

整體來看，FF hold 是改善最明顯的部分。testcase1、testcase2、testcase3 的 FF WNS/TNS/NVP 都修到 0，代表 hold repair 與方向式 skew 調整有效。testcase0 的 FF TNS 從 -3.0381 改善到 -0.4664，FF WNS 從 -0.1085 改善到 -0.0279，也有明顯進步；testcase4 因規模最大、原始 violation 最多，仍保留較多 FF violation，但 FF_TNS_term 與 FF_WNS_term 仍分別達到 0.5696 與 0.4668。

Setup timing 方面，所有 testcase 的 SS TNS/WNS 都有改善，但 SS WNS 仍常是瓶頸，尤其 testcase4 的 SS_WNS_term 只有 0.2073。這表示目前演算法可以減少大量 violating paths 或降低總 violation，但要精準修到最差單一路徑仍較困難。原因可能是 worst setup path 需要的 skew 改變較集中，若只靠 buffer chain 或 resize，會受到 subtree 共用、fanout、hold 保護條件與 area 的限制。

Area 結果呈現取捨。testcase1、testcase2、testcase3 分別降低 8.60%、18.27%、2.50%，顯示 direction-aware downsize 與 safe cleanup 對中小型 testcase 有幫助。testcase0 與 testcase4 則小幅增加 0.05% 與 0.07%，表示在較大且 violation 較嚴重的案例中，為了修 timing 仍需要插入額外 buffer。這個 tradeoff 是合理的，因為 timing terms 在評分中比 area 更優先，程式也刻意把 AREA_term 當成 penalty / cleanup target，而不是早期主要目標。

Runtime 方面，`optimization_notes.md` 記錄 testcase0 solver wall time 約 570.72 sec，testcase2 約 552.84 sec；目前 final code 中 `kOptimizationTimeLimitSec` 為 89 sec，代表不同實驗版本使用過不同時間設定。正式報告若要呈現最終 runtime，建議以實際繳交版本重新跑一次 checker 與 solver，補上每個 testcase 的精確 wall time。
